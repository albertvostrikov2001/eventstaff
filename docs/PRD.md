# PRD — платформа «Юнити» (полная версия)

| Поле | Значение |
|------|----------|
| **Версия документа** | 2.0 |
| **Статус** | По состоянию кодовой базы репозитория (монорепозиторий `web` + `api` + `shared`) |
| **Язык интерфейса** | Русский (основной) |

Документ описывает **что должен уметь продукт** в терминах бизнеса и пользователей. Технические детали API приведены как справочный перечень для согласования скоупа с разработкой.

---

## 1. Резюме для руководства

**Юнити** — онлайн-биржа **event-персонала** для ресторанов, кейтеринга, отелей и event-индустрии: работодатели размещают потребность в людях, специалисты откликаются и проходят согласование смен; платформа даёт **профили, каталоги, коммуникации, учёт смен, репутацию, модерацию и администрирование**.

Ценность: сокращение времени подбора, единая точка коммуникации, задел под монетизацию (платежи, услуги платформы).

## 5. Область продукта

### 5.1 Входит в скоуп (реализуется системой)

- Публичный маркетинговый сайт и каталоги
- Регистрация, вход, сессии, сброс пароля
- Кабинеты работника и работодателя
- Вакансии и отклики, избранное, приглашения работодателя
- Бронирования (**Booking**) и **смены (Shift)** с подтверждением сторон, отменами, сбоем, спором
- Сообщения (**Conversation/Message**) и чат в реальном времени (**Socket**)
- Отзывы по сменам (**ShiftReview**) и модель надёжности (**UserReliabilityScore**)
- Жалобы (**Complaint**) и публичные репорты (**Report**) в данных
- Медиафайлы с модерацией (**Media**)
- Внутриплатформенные уведомления и **email** (очередь, шаблоны, логи)
- Платежи (ЮKassa и расширяемость), платежи по сменам в модели данных
- Админ-панель: пользователи, вакансии, отклики, жалобы, медиа, письма, аудит, индивидуальные заявки, спорные смены
- Публичная форма **индивидуальной заявки** (подбор персонала под мероприятие)
- Справочник городов, типы бизнеса и категории персонала

### 5.2 Ограничения и «частично готово»

- **Подписки работодателя** (`Subscription`): таблицы и связи есть, **полная тарифная логика и лимиты** — отдельный этап
- **Шаблоны вакансий** (`VacancyTemplate`): модель есть, **отдельный CRUD в API** не заявлен в коде как завершённый (`BACKLOG.md`)
- **Доступность работника** (`WorkerAvailability`): хранится, **автоблокировка слотов при брони** — в backlog (не связано автоматически)
- **OAuth / соцвход**: в README упомянут, **в текущих роутах авторизации не реализован**
- **SMS OTP**: утилиты/задел возможен, **сквозной сценарий верификации телефона** — не заявлен
- **Push / Telegram** уведомления: поля/типы в данных, **доставка** — не заявлена
- **SEO**: динамические meta для списков каталога — в backlog

## 6. Роли и права

### 6.1 Роли в системе

| Роль | Код | Кабинет (веб) |
|------|-----|----------------|
| Специалист | `worker` | `/worker/*` |
| Работодатель | `employer` | `/employer/*` |
| Администратор | `admin` | `/admin/*` |
| Модератор | `moderator` | По политике внедрения (в коде — уточнение прав отдельно от `admin`) |

Пользователь может иметь **несколько ролей**; активная роль влияет на отображение и действия. Защита маршрутов на фронте (middleware) и **полная проверка на API** обязательны.

### 6.2 Статусы учётной записи

| Статус | Последствия для пользователя |
|--------|-------------------------------|
| `active` | Нормальная работа |
| `restricted` | Ограничения по политике (например, частичный доступ) |
| `banned` | Блокировка (нельзя пользоваться сервисом) |
| `deleted` | Удалённая учётная запись |

## 8. Пользовательские потоки (end-to-end)

### 8.1 Регистрация и вход

1. Пользователь выбирает роль (работник / работодатель), задаёт учётные данные  
2. После входа получает доступ к соответствующему кабинету  
3. **Сброс пароля**: запрос по email → письмо → установка нового пароля  
4. Сессия: **access/refresh** через HTTP-only cookies; при 401 клиент обновляет сессию или перенаправляет на вход  

**Ограничение:** частота регистрации ограничена rate limit на API.

### 8.2 Работодатель: вакансия и найм

1. Создаёт/редактирует вакансию (ставка, тип занятости, мероприятие, требования)  
2. Вакансия проходит статусы: черновик → **модерация** (при необходимости) → активна / отклонена / архив  
3. Просматривает отклики, меняет статусы (просмотрен, приглашение, отказ и т.д.)  
4. Может пригласить работника **напрямую** (по профилю)  
5. Формируется **бронь** и **смена**; стороны подтверждают выполнение  
6. После смены возможен **отзыв** и обновление **надёжности**  
7. При конфликте — **жалоба** и/или статус **спора** у смены для админа  

### 8.3 Специалист: отклик и работа

1. Заполняет профиль, категории, медиа (после одобрения — в каталоге при разрешённой видимости)  
2. Ищет вакансии в каталоге, откликается с сообщением  
3. Видит статус отклика, приглашения работодателя  
4. Управляет **доступностью по датам** (календарь; автосвязь с бронью — см. backlog)  
5. Подтверждает смену, завершает, оставляет отзыв  
6. Использует **избранные вакансии**  
7. Настраивает **уведомления** (email-каналы по типам событий)  

### 8.4 Коммуникации

- Классический **список переписок** и история сообщений (REST)  
- **Чат** в реальном времени: комнаты, сообщения, счётчик непрочитанных (WebSocket)  

### 8.5 Публичный сайт

- Главная, «для работодателей/специалистов», **тарифы**, контакты, справка  
- Юридические страницы: оферта, конфиденциальность, условия  
- Каталог **специалистов** и **вакансий** с фильтрами  
- Карточка работодателя по **slug**  
- Страница **заявки** на индивидуальный подбор (`/request`) → создаётся **IndividualRequest**  

### 8.6 Администрирование

- Поиск и управление пользователями: **бан**, **снятие ограничений**, смена ролей (в рамках политики)  
- Верификация работодателя; управление **видимостью** профиля работника  
- Очередь **модерации медиа** (одобрить / отклонить)  
- **Жалобы**: просмотр, смена статуса, история действий админа  
- **Спорные смены**: просмотр, разрешение спора  
- **Индивидуальные заявки**: очередь, комментарий админа, статус  
- **Email logs**: отправлено/ошибка, повтор при сбое (где реализовано)  
- **Аудит** действий администраторов  
- Сводка дашборда (агрегированные метрики)

## 9. Детальные функциональные требования

### 9.1 Каталог (публичный API)

- Список городов  
- **Специалисты**: пагинация, фильтры (город, категория и др. по реализации), карточка  
- **Вакансии**: фильтры, карточка, счётчик просмотров (где включено)  
- **Работодатели**: список и публичная карточка  

**Требование:** не раскрывать персональные контакты сверх публичной политики видимости профиля.

### 9.2 Профиль работника

- Поля: ФИО, возраст, город, био, опыт, ставка и тип ставки, языки, медкнижка (флаг), готовность к разъездам/переработке, **видимость** (`public` / `verified_only` / `hidden` / `invite_only`)  
- **Категории** персонала: добавление/удаление, уровень, специализация  
- **Портфолио** и документы (в т.ч. со статусом проверки документов)  
- История работ (опционально, внешние места)  

### 9.3 Профиль работодателя

- Тип: компания / индивидуальный  
- Поля: название, контакт, описание, сайт, ИНН (опционально), город, тип бизнеса  
- Медиа: логотип, баннер, галерея (через общий модуль **Media**)  
- Флаги верификации и запроса верификации  

### 9.4 Вакансии и отклики

**Статусы вакансии:** черновик, на модерации, активна, закрыта, архив, отклонена  

**Статусы отклика (примерный набор):** ожидает, просмотрен, приглашение, собеседование, подтверждён, смена начата, завершён, отменён, отклонён  

**Требования:**  
- Один отклик работника на вакансию (уникальная пара)  
- История изменения статуса сохраняется (для разбора споров)  
- Работодатель меняет статус отклика с валидацией прав  

### 9.5 Бронирования и смены

- **Booking** связывает работодателя и работника, опционально — исходную вакансию  
- **Shift** однозначно привязан к брони и участникам  
- Жизненный цикл: ожидание → активна → подтверждения → завершена / отменена / сорвана / **спор**  
- Подтверждение «смена состоялась» — обеими сторонами (флаги и временные метки)  
- **Срыв** с указанием стороны и причины (коды причин для работника/работодателя)  
- **Отзыв по смене** в окне времени (в коде задано окно, например 72 ч — для согласования с бизнесом)  
- Эскалация «зависших» подтверждений — фоновая обработка + ручной запуск админом (endpoint)  

### 9.6 Репутация и надёжность

- У пользователя есть агрегированный **балл надёжности** и **уровень** (NEW → … → RESTRICTED)  
- Учитываются успешные/сорванные/отменённые смены, **страйки**, ограничения  

### 9.7 Жалобы

- Типы: пользователь, сообщение, вакансия, заказ, отмена, неоплата, неявка и др.  
- Статусы: новая, в работе, решена, отклонена, нужны уточнения  
- История действий админа фиксируется  

### 9.8 Медиа

- Загрузка файлов (multipart), тип: аватар, портфолио, логотип, баннер, галерея, документ, видео-карточка и т.д.  
- Модерация: до одобрения публичное использование ограничено политикой **Media**  
- Удаление своих файлов пользователем (где разрешено)  

### 9.9 Уведомления

**In-app:** типы из перечисления (приглашение, отклик, отмена, отзыв, жалоба, смена, оплата, заявка, система, сброс пароля, …)  

**Email:**  
- Очередь через worker (BullMQ/Redis)  
- Лог в **EmailLog** (ожидает/отправлено/ошибка)  
- Настройки пользователя: какие email включены (приглашения, отмены, отзывы, жалобы, новые отклики, ответы по отклику)  

### 9.10 Платежи

- Создание платежа работодателем (интеграция **ЮKassa** в коде)  
- Статусы платежа, метаданные  
- Платежи, связанные со **сменами** (отдельная модель **ShiftPayment**) — для учёта выплат за смену  

### 9.11 Избранное

- Работник: избранные **вакансии**  
- Работодатель: избранные **работники**

## 10. Справочник API (группы, без полного OpenAPI)

Базовый префикс: **`/api/v1`**.

| Группа | Префикс / путь | Назначение |
|--------|----------------|------------|
| Здоровье | `/health` | Проверка живости |
| Авторизация | `/auth/*` | Регистрация, вход, выход, refresh, `/me`, forgot/reset password |
| Работник | `/worker/*` | Профиль, категории, отклики, избранное, доступность, смены (список) |
| Работодатель | `/employer/*` | Профиль, вакансии CRUD, статусы откликов, приглашения, избранное, смены, приглашения-лист |
| Каталог | `/catalog/*` | Города, работники, вакансии, работодатели |
| Сообщения | `/messages/*` | Диалоги и сообщения |
| Чат | `/chat/*` | Комнаты, сообщения, непрочитанное |
| Медиа | `/media/*` | Загрузка, свои файлы, удаление |
| Уведомления | `/notifications/*` | Список, read-all, настройки |
| Платежи | `/payments/*` | Создание платежа, статусы, webhook (секрет) |
| Общее | `/complaints`, `/reliability`, `/users/:id/reviews`, `/shifts/:id`, отмена заказа, **individual-requests** | Жалобы, надёжность, отзывы, просмотр смены, индивидуальные заявки |
| Смены (действия) | `/shifts/:id/*` | Подтверждение, срыв, отзыв; админ: эскалация |
| Админ | `/admin/*` | Пользователи, модерация, жалобы, медиа, логи почты, аудит, заявки, спорные смены, статистика |

Полная интерактивная спецификация: **Swagger UI** на `/docs` у запущенного API.

## 16. Критерии приёмки (шаблон эпика)

Для каждой крупной функции заказчик и исполнитель фиксируют:

1. **Дано** — роль и предусловия в БД  
2. **Когда** — действие пользователя или вызов API  
3. **Тогда** — ожидаемый результат в UI и в БД (статусы, записи логов)  
4. **Ошибки** — понятные сообщения, отсутствие утечек данных  

**Минимальный сквозной тест перед сдачей:** регистрация двух ролей → вакансия → отклик → переписка → смена → отзыв → просмотр админом.
